Esta solução fornece uma visão geral e detalhes para habilitar o Oracle High Availability (HA) no AWS EC2 com clustering de pacemaker no Redhat Enterprise Linux (RHEL) e Amazon FSX ONTAP para armazenamento de banco de dados HA via protocolo NFS.



== Finalidade

Muitos clientes que se esforçam para autogestionar e executar o Oracle na nuvem pública precisam superar alguns desafios. Um desses desafios é permitir alta disponibilidade para o banco de dados Oracle. Tradicionalmente, os clientes Oracle dependem de um recurso de banco de dados Oracle chamado "Real Application Cluster" ou RAC para suporte a transações ativo-ativo em vários nós de cluster. Um nó com falha não interromperia o processamento de aplicativos. Infelizmente, a implementação do Oracle RAC não está prontamente disponível ou suportada em muitas nuvens públicas populares, como o AWS EC2. Ao aproveitar o cluster de pacemaker (PCS) integrado no RHEL e no Amazon FSX ONTAP, os clientes podem obter uma alternativa viável sem o custo de licença do Oracle RAC para clustering ativo-passivo em computação e armazenamento para suportar a carga de trabalho de banco de dados Oracle de missão crítica na nuvem da AWS.

Esta documentação demonstra os detalhes da configuração de cluster de pacemaker no RHEL, implantação de banco de dados Oracle no EC2 e Amazon FSX ONTAP com protocolo NFS, configuração de recursos Oracle no pacemaker para HA e finalização da demonstração com validação na maioria dos cenários de HA encontrados. A solução também fornece informações sobre o rápido backup, restauração e clone de banco de dados Oracle com a ferramenta de IU do NetApp SnapCenter.

Esta solução aborda os seguintes casos de uso:

* Configuração e configuração de cluster do pacemaker HA no RHEL.
* Implantação de HA de banco de dados Oracle no AWS EC2 e no Amazon FSX ONTAP.

=== Fatores-chave para consideração da implantação

* *Amazon FSX ONTAP HA.* O Amazon FSX ONTAP é provisionado em um par de controladores de armazenamento de HA em uma ou várias zonas de disponibilidade por padrão. Ele fornece redundância de storage de forma ativa/passiva para cargas de trabalho de banco de dados de missão crítica. O failover de storage é transparente para o usuário final. A intervenção do usuário não é necessária no caso de um failover de armazenamento.
* * Grupo de recursos DO PCS e pedidos de recursos.* Um grupo de recursos permite que vários recursos com dependência sejam executados no mesmo nó de cluster. A ordem de recurso impõe a ordem de inicialização de recursos e a ordem de desligamento em sentido inverso.
* *Nó preferido.* O cluster de pacemaker é propositadamente implantado em clustering ativo/passivo (não um requisito do pacemaker) e está em sincronia com o cluster FSX ONTAP. A instância EC2 ativa é configurada como um nó preferencial para recursos Oracle quando disponível com uma restrição de local.
* * Atraso de cerca no nó de espera.* Em um CLUSTER DE PCS de dois nós, um quórum é definido artificialmente como 1. No caso de um problema de comunicação entre os nós do cluster, qualquer nó pode tentar cercar o outro nó, o que pode potencialmente causar corrupção de dados. A configuração de um atraso no nó de espera atenua o problema e permite que o nó principal continue fornecendo serviços enquanto o nó de espera está vedado.
* *Consideração de implantação Multi az.* A solução é implantada e validada em uma única zona de disponibilidade. Para a implantação multi-az, são necessários recursos adicionais de rede da AWS para mover o IP flutuante DOS PCS entre as zonas de disponibilidade.
* * Layout de armazenamento de banco de dados Oracle.* Nesta demonstração de solução, provisionamos três volumes de banco de dados para o banco de dados de teste NTAP para hospedar binários, dados e log Oracle. Os volumes são montados no servidor Oracle DB como /u01 - binário, /u02 - dados e /u03 - log via NFS. Os arquivos de controle duplo são configurados em pontos de montagem /u02 e /u03 para redundância.
* *Configuração DNFS.* Ao usar o DNFS (disponível desde o Oracle 11gi), um banco de dados Oracle executado em uma VM de banco de dados pode gerar significativamente mais I/o do que o cliente NFS nativo. A implantação automatizada do Oracle configura o DNFS no NFSv3 por padrão.
* *Backup do banco de dados.* O NetApp fornece um pacote de software SnapCenter para backup, restauração e clonagem de banco de dados com uma interface de usuário amigável. A NetApp recomenda a implementação dessa ferramenta de gerenciamento para obter backup instantâneo rápido (em menos de um minuto), restauração rápida de banco de dados (em minutos) e clone de banco de dados.

=== Configuração de esgrima do cluster do pacemaker

[%collapsible%open]
====
A configuração de esgrima do pacemaker é obrigatória para um cluster de produção. Ele garante que um nó com defeito no cluster do AWS EC2 seja isolado automaticamente, impedindo que o nó consuma os recursos do cluster, comprometa a funcionalidade do cluster ou corrompe os dados compartilhados. Esta seção demonstra a configuração de cercas de cluster usando o agente de cercas fence_aws.

. Como usuário raiz, insira a seguinte consulta de metadados da AWS para obter o ID da instância para cada nó de instância do EC2.
+
[source, cli]
----
echo $(curl -s http://169.254.169.254/latest/meta-data/instance-id)
----
+
....
[root@ip-172-30-15-111 ec2-user]# echo $(curl -s http://169.254.169.254/latest/meta-data/instance-id)
i-0d8e7a0028371636f

or just get instance-id from AWS EC2 console
....
. Digite o seguinte comando para configurar o dispositivo de vedação. Use o comando pcmk_host_map para mapear o nome do host RHEL para o ID da instância. Use a chave de acesso da AWS e a chave de acesso secreta da AWS da conta de usuário da AWS que você usou anteriormente para autenticação da AWS.
+
[source, cli]
----
sudo pcs stonith \
create clusterfence fence_aws access_key=XXXXXXXXXXXXXXXXX secret_key=XXXXXXXXXXXXXXXXXX \
region=us-east-1 pcmk_host_map="ip-172-30-15-111.ec2.internal:i-0d8e7a0028371636f;ip-172-30-15-5.ec2.internal:i-0bc54b315afb20a2e" \
power_timeout=240 pcmk_reboot_timeout=480 pcmk_reboot_retries=4
----

=== Implantar banco de dados Oracle no cluster PCS

[%collapsible%open]
====
Recomendamos que utilize o manual de estratégia do Ansible fornecido pela NetApp para executar tarefas de instalação e configuração de banco de dados com parâmetros predefinidos no cluster DE PCS. Para essa implantação automatizada da Oracle, três arquivos de parâmetros definidos pelo usuário precisam de entrada do usuário antes da execução do manual de estratégia.

* hosts - defina os destinos com os quais o manual de estratégia de automação está sendo executado.
* vars/vars.yml - o arquivo de variável global que define variáveis que se aplicam a todos os destinos.
* host_vars/host_name.yml - o arquivo de variável local que define variáveis que se aplicam somente a um destino nomeado. No nosso caso de uso, estes são os servidores Oracle DB.


Além desses arquivos de variáveis definidos pelo usuário, existem vários arquivos de variáveis padrão que contêm parâmetros padrão que não exigem alteração, a menos que necessário.
