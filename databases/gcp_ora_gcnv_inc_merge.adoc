:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs:

NetApp

[role="lead"]
A solução fornece uma visão geral e detalhes para recuperação rápida do Oracle VLDB implantado na VM do Google Compute Engine com montagem NFS no pool de armazenamento do Google Cloud NetApp Volumes para preparar uma cópia do banco de dados em espera que é mesclada de forma incremental e constante via RMAN.



== Finalidade

Recuperar um banco de dados muito grande (VLDB) no Oracle usando a ferramenta de backup do Oracle Recovery Manager (RMAN) pode ser uma tarefa altamente desafiadora. O processo de restauração de banco de dados a partir de Mídia de backup em caso de falha pode ser demorado, atrasando a recuperação do banco de dados e potencialmente impactando significativamente o SLA (Service Level Agreement). A situação está piorando ainda mais para o Oracle VLDB implantado na nuvem, já que o backup do RMAN geralmente é transmitido para o armazenamento de objetos devido a considerações de custo. Como resultado, não é viável restaurar e recuperar um VLDB se o SLA for essencial. A partir da versão 10g, a Oracle introduziu um recurso do RMAN que permite aos usuários criar cópias de imagem em estágios dos arquivos de dados do banco de dados Oracle em armazenamento em disco adicional localizado no host do servidor do banco de dados. Essas cópias de imagem podem ser atualizadas incrementalmente usando o RMAN em um intervalo definido pelo usuário. Em caso de falha, o Administrador de Banco de dados (DBA) pode mudar rapidamente o banco de dados Oracle da Mídia com falha para a cópia de imagem, eliminando a necessidade de uma restauração completa de Mídia de banco de dados. O resultado é um SLA bastante aprimorado, embora com o custo de dobrar o storage necessário do banco de dados.

Se você está interessado em SLA para seu VLDB e está pensando em mover o banco de dados Oracle para uma nuvem pública, como o Google Cloud Platform (GCP), pode configurar uma estrutura de proteção de banco de dados semelhante usando recursos como o Google Cloud NetApp Volumes (GCNV) para preparar sua cópia de imagem do banco de dados standby. O GCNV é um serviço de armazenamento de arquivos de alto desempenho e totalmente gerenciado, oferecido pelo Google, que torna mais fácil, seguro e econômico migrar e executar aplicativos corporativos exigentes, como o banco de dados Oracle, no Google Cloud. Seus variados níveis de serviço permitem a flexibilidade de controlar o custo de armazenamento, ao mesmo tempo em que atinge um RPO próximo de zero e um RTO gerenciável. Nesta documentação, demonstramos como provisionar e exportar um sistema de arquivos NFS de um pool de armazenamento GCNV para ser montado em um servidor de banco de dados Oracle para preparar uma cópia do banco de dados standby para recuperação rápida em caso de falha do armazenamento primário.

Esta solução aborda os seguintes casos de uso:

* Backup de cópia de imagem de arquivos de dados do banco de dados Oracle via RMAN para um sistema de arquivos NFS no servidor em um pool de armazenamento do Google Cloud NetApp Volumes.
* Recuperação rápida de um Oracle VLDB em caso de falha no mesmo ou em uma VM do Google Compute Engine em espera.
* Clone uma cópia do banco de dados Oracle em um host clone para outros casos de uso a partir da cópia da imagem do banco de dados de preparação em minutos.




== Público-alvo

Esta solução destina-se às seguintes pessoas:

* Um DBA que configura e gerencia o Oracle VLDB no Google Cloud Platform.
* Um arquiteto de soluções de banco de dados que testa cargas de trabalho Oracle no Google Cloud Platform.
* Um administrador de armazenamento que gerencia bancos de dados Oracle implantados em pools de armazenamento GCNV.
* Um proprietário de aplicativo que gostaria de instalar bancos de dados Oracle no Google Cloud Platform.

=== Fatores-chave para consideração da implantação

* *Nível de serviço do pool de armazenamento GCNV.* O pool de armazenamento GCNV oferece quatro níveis de serviço: Standard, Premium, Extreme e Flex. Por padrão, um QoS automático é aplicado a um volume criado dentro de um pool de armazenamento, o que restringe a taxa de transferência do volume com base em seu tamanho e no nível de serviço. A única exceção é que os volumes alocados do pool de armazenamento Flex compartilham a taxa de transferência disponível para o pool de armazenamento.
* *Limite de armazenamento de backup no servidor Oracle VLDB para mesclagem incremental RMAN.* Em nosso teste e validação, um volume NFS para backup e mesclagem incremental Oracle é alocado a partir de um único pool de armazenamento GCNV com nível de serviço padrão, que tem um limite de capacidade de 200 TiB. Para implantações acima desses limites, vários pools de armazenamento GCNV podem ser implementados em paralelo com vários pontos de montagem NFS para fornecer maior capacidade.
* *Recuperação Oracle usando a mesclagem incremental RMAN.* O backup incremental e a mesclagem no servidor RMAN geralmente são executados em uma frequência definida pelo usuário, com base nos seus objetivos de RTO e RPO. Se houver perda total do armazenamento de arquivos do banco de dados primário e/ou dos logs arquivados, poderá ocorrer perda de dados. O banco de dados Oracle pode ser recuperado até o último backup incremental disponível a partir da cópia da imagem de backup do banco de dados no servidor. Para minimizar a perda de dados, a área de recuperação flash do Oracle precisa ser configurada no ponto de montagem do backup NFS GCNV, e os logs arquivados são copiados para a mesma montagem NFS, juntamente com a cópia da imagem do banco de dados, para que o banco de dados avance para o último log arquivado, em caso de falha. Assim, é possível atingir um RPO próximo de zero.
* *Executando Oracle VLDB a partir de backup de imagem NFS no pool de armazenamento GCNV.* Neste momento, não há suporte para reatribuir o nível de serviço do volume do banco de dados de padrão para premium ou extremo. No entanto, a reatribuição de níveis de serviço de premium para extremo é suportada. Para executar o banco de dados em uma cópia de imagem por um período prolongado após uma falha, é recomendável alocar volume do pool de armazenamento premium para backup da cópia de imagem do banco de dados.
* *VMs do Google Compute Engine.* No teste e na validação, usamos uma instância de VM do GCE do tipo N1 como servidores de banco de dados Oracle. Existem outros tipos de VMs do GCE que podem ser mais otimizados e adequados para a carga de trabalho do banco de dados. Dimensione a VM adequadamente para o número de vCPUs e a quantidade de RAM, com base nos requisitos reais da carga de trabalho.
* *Configuração DNFS* o DNFS é incorporado ao kernel Oracle e é conhecido por aumentar drasticamente o desempenho do banco de dados Oracle quando o Oracle é implantado no armazenamento NFS. O DNFS é empacotado no binário Oracle, mas não é ativado por padrão. Ele deve ser ativado para qualquer implantação de banco de dados Oracle no NFS. Para a implantação de vários pools de armazenamento GCNV para um VLDB, os múltiplos caminhos dNFS para diferentes pools de armazenamento GCNV devem ser configurados corretamente.




== Implantação de solução

A solução pressupõe que um Oracle VLDB já esteja implantado no ambiente do Google Cloud dentro de uma VPC. O Oracle VLDB pode ser executado no armazenamento GCNV ou em qualquer outra opção de armazenamento dentro do ecossistema do Google Cloud. A seção a seguir fornece procedimentos de implantação passo a passo para configurar um backup de imagem do Oracle VLDB e mesclagem incremental via RMAN para um sistema de arquivos NFS montado a partir de um pool de armazenamento GCNV. O Oracle VLDB também tem seu armazenamento primário hospedado em um pool de armazenamento GCNV diferente.

=== Monte um volume NFS do GCNV para o servidor Oracle VLDB primário

[%collapsible%open]
====
O volume GCNV provisionado para o backup no servidor Oracle VLDB deve, idealmente, estar situado na mesma zona do Google Cloud que o servidor de banco de dados principal e o armazenamento principal para melhor desempenho.

. Recupere o caminho de exportação NFS do volume GCNV do console do Google Cloud.
+
image:gcp_ora_gcnv_inc_merge_vol_03.png["Esta imagem mostra a tela de volumes de armazenamento GCNV do console do portal do Google Cloud"]

. Efetue login no servidor Oracle VLDB primário. Como root, monte o volume NFS exportado do armazenamento GCNV. Altere o endereço IP e o caminho do arquivo do seu servidor NFS conforme necessário.
+
[source, cli]
----
sudo mkdir /nfsgcnv
----
+
[source, cli]
----
mount 10.165.128.5:/orap-bkup /nfsgcnv -o rw,bg,hard,vers=3,proto=tcp,timeo=600,rsize=262144,wsize=262144
----
. Altere a propriedade do ponto de montagem para oracle:oisntall ou altere seu nome de usuário e grupo primário do Oracle, conforme necessário.
+
[source, cli]
----
sudo chown oracle:oinstall /nfsgcnv
----
. Adicione a montagem NFS em /etc/fstab.
+
[source, cli]
----
sudo vi /etc/fstab
----

=== Configurar mesclagem incremental do Oracle RMAN para cópia de imagem no GCNV para backup VLDB no servidor

[%collapsible%open]
====

A cópia de imagem da cópia de segurança da base de dados será tão atualizada quanto a frequência que executa a cópia de segurança/fusão incremental. Portanto, leve em consideração o desempenho do banco de dados, seus objetivos de rto e RPO ao decidir a frequência de backup e mesclagem incrementais do RMAN.

. Crie um script de backup e mesclagem incremental RMAN. O script aloca vários canais para backup e mesclagem RMAN paralelos. A primeira execução geraria a cópia inicial da imagem de linha de base completa. Em uma execução completa, ele primeiro limpa backups obsoletos que estão fora da janela de retenção para manter a área de teste limpa. Em seguida, ele alterna o arquivo de log atual antes de mesclar e fazer backup. O backup incremental segue a mesclagem de modo que a cópia de imagem do banco de dados esteja perdendo o estado atual do banco de dados por um ciclo de backup/mesclagem. A ordem de mesclagem e backup pode ser revertida para uma recuperação mais rápida de acordo com a preferência do usuário. O script RMAN pode ser integrado em um script shell simples para ser executado a partir do crontab no servidor de banco de dados primário. Certifique-se de que o backup automático do arquivo de controle esteja ativado na configuração RMAN.

=== Recuperação rápida do Oracle VLDB usando cópia de imagem de backup



==== Recuperação completa de perda ou corrupção de arquivos de dados Oracle

[%collapsible%open]
====
No caso de uma falha devido a problemas de storage primário, como perda de dados ou corrupção, o banco de dados pode ser rapidamente comutado para cópia de imagem na montagem de GCNV NFS e recuperado para o estado atual sem restauração de banco de dados. A eliminação da restauração de Mídia acelera tremendamente a recuperação do banco de dados para um VLDB. Este caso de uso pressupõe que o servidor Oracle VLDB DB está intacto e o arquivo de controle de banco de dados, os logs arquivados e atuais estão todos disponíveis para recuperação.

==== Recuperação pontual com perda de todos os arquivos Oracle

[%collapsible%open]
====
Em caso de falha, onde o armazenamento primário e/ou o host do servidor de banco de dados primário são perdidos, todas as transações que não estão nos arquivos de log arquivados serão perdidas. No entanto, sua cópia de imagem de backup do banco de dados Oracle disponível no sistema de arquivos GCNV NFS é útil. Você pode recuperar rapidamente o banco de dados primário para o mesmo servidor de banco de dados primário ou para um servidor de banco de dados em espera, se houver um disponível, usando a cópia da imagem de backup. Nesta seção, demonstraremos os procedimentos passo a passo para a recuperação para o servidor de banco de dados primário. Os métodos de recuperação para um novo servidor de banco de dados em espera são quase idênticos; no entanto, a cópia da imagem de backup do banco de dados deve ser montada no servidor de banco de dados em espera com a mesma pilha de software Oracle instalada e configurada.

. Para retornar ao armazenamento primário, siga exatamente os mesmos procedimentos detalhados na seção  `Reverse VLDB back to primary storage from backup copy` . Se a recuperação estiver sendo feita em um servidor de banco de dados em espera, o armazenamento primário precisa ser importado para o servidor de banco de dados em espera. Também é necessário realocar os arquivos de controle do Oracle e os arquivos de log online atuais da montagem GCNV NFS de volta para o armazenamento primário, o que pode ser feito em poucos minutos de inatividade.


Isso conclui a recuperação pontual do banco de dados Oracle VLDB da cópia da imagem de backup no sistema de arquivos GCNV NFS para o mesmo servidor de banco de dados primário ou um servidor de banco de dados em espera.

====
